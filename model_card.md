# Model Card — VibeFinder 2.0

## Project identity

**Project name:** VibeFinder 2.0 — AI Music Recommender with Reliability Evaluation
**Base project:** VibeFinder 1.0 (Module 3 — Music Recommender Simulation)
**Extended AI feature:** Reliability testing system — confidence scoring, guardrails, evaluator module, and automated test harness
**Model used:** Claude (claude-opus-4-5) via Anthropic API — used exclusively for natural language → preference extraction

---

## What the system does

VibeFinder 2.0 wraps the original content-based music recommender in three new reliability layers:

1. **Guardrail layer** — screens input for prompt injection and role-override patterns before any API call is made.
2. **AI Preference Parser** — sends the user's natural language description to the Anthropic API and receives a structured JSON preference object plus a self-confidence score.
3. **Evaluator** — after recommendations are generated by the original engine, it scores the result set for genre match rate, mood match rate, and average quality — producing a `quality_score` (0.0–1.0) that is combined with the AI confidence to show a final reliability indicator.

---

## Limitations and biases

- **Small song catalog.** The dataset has 50 songs covering 10 genres. Recommendations for niche genres (e.g. "bossa nova", "ambient") may be inaccurate because those genres are not represented.
- **Genre and mood taxonomy is fixed.** The AI parser is instructed to map all inputs to one of 10 genres and 10 moods. Nuanced descriptions ("dreamy shoegaze", "dark jazz") get forced into the closest available bucket, which may not match the user's intent.
- **Self-reported confidence is not calibrated.** The model estimates its own certainty, but this is not derived from probability distributions. It may be overconfident on imprecise extractions or underconfident on correct ones.
- **Keyword-based guardrails can be bypassed.** A user who rephrases an injection attempt in unusual language may pass the filter. A semantic classifier would be more robust.
- **English-language bias.** The system prompt and song data are in English. Non-English inputs may produce lower-quality extractions.

---

## Potential for misuse and mitigations

| Risk | Mitigation |
|---|---|
| Prompt injection to override system behavior | Guardrail layer blocks known patterns before API call |
| User trusting low-confidence output as authoritative | Confidence bar + flag for results below 0.6 |
| Using the AI to generate harmful content via music framing | System prompt restricts output to JSON preference extraction only; Anthropic safety filters apply at API level |
| Log files containing sensitive user queries | logs/ directory is excluded from git via .gitignore |

**What surprised me:** Confidence scores from the AI parser were less stable than expected across repeated runs with identical inputs. The same vague phrase ("I want something chill") could return confidence 0.65 or 0.80 on different calls. This is an inherent property of LLM output and required changing the test assertions from exact-value checks to range checks — a fundamentally different approach from testing deterministic code.

## AI collaboration reflection

### Instance where the AI gave a helpful suggestion

When designing the evaluator, the AI (Claude) suggested weighting genre match rate at 0.40 and mood match rate at 0.30, arguing that genre is a harder constraint for users (they almost always have a firm genre preference) while mood is softer and more flexible. This matched my own intuition but articulated it more clearly. I adopted this weighting and it produced sensible quality scores across all test profiles.

### Instance where the AI's suggestion was flawed

The AI initially suggested returning `None` for the confidence score when the AI parser returned a malformed JSON response, and then checking for `None` downstream before computing the combined score. In practice, this meant every downstream function needed an explicit None-check, which made the code brittle and led to a `TypeError` during early testing when `confidence_bar(None)` was called. The correct fix was to default to a neutral `0.8` fallback so all downstream functions always receive a float. The AI prioritized semantic correctness (None communicates "unknown") over practical robustness (a float never crashes downstream code). The right answer was the opposite trade-off.

## What this project taught me about AI and problem-solving

The most important lesson was the difference between a system that *works* and a system that *communicates honestly*. VibeFinder 1.0 always returned 5 songs — it had no way to say "I'm not sure these are right for you." Adding the confidence indicator changed the user's relationship with the output. A 42% confidence score on a vague request is not a failure; it is the system being honest, which is more valuable than false precision.

The second lesson was about testing probabilistic systems. Every unit test I had written before assumed deterministic outputs. Testing AI components required a new mental model: define acceptable ranges, test structural properties (keys present, types correct, order maintained), and treat exact output content as inherently variable. This shift in thinking — from "does it return X?" to "does it behave correctly?" — is the core skill for AI engineering.
